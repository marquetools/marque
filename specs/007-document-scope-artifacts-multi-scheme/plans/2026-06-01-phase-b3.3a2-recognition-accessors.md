# Phase B3.3a.2 — recognition-accessor trait-lift (addendum to B3.3a)

## Why an addendum

B3.3a (#839) lifted the engine's CAPCO-inherent **canonical-space** operations onto
trait surface (`canonical_page_join`, `project_canonical[_with_sink]`,
`marking_from_canonical`) plus the `ConstraintBridge` constraint→diagnostic hooks, so
B3.3b can drive the page-accumulation / projection / bridge layers generically.

Scoping B3.3b revealed that B3.3a was **under-provisioned**: the engine's
recognition/reparse boundary reaches across three *more* CAPCO-specific reads that have
no trait surface, so `impl<S: MarkingScheme + ConstraintBridge> Engine<S, R>` cannot
compile without them. The lint loop is one monolithic function, so the recognition step
cannot stay `CapcoScheme`-pinned while the rest generifies.

| Inherent read (today)                                            | Where | New trait surface |
|------------------------------------------------------------------|-------|-------------------|
| `let attrs = marking.0` (recognition); `marking.0.clone()` (fix reparse cache) | `lint_helpers::recognize_marking_candidate`, `fix_impl::populate_pre_pass_1_cache` | `MarkingScheme::canonical_from_marking` |
| `attrs.classification.effective_level() as u8` (rank floor)      | `recognize_marking_candidate` | `MarkingScheme::canonical_rank` |
| `marking.1` (decoder provenance): is-decoder-path / score / `build_decoder_diagnostic` | `recognize_marking_candidate` | `ConstraintBridge::recognition_outcome` |

This mirrors B3.3a's own sequencing note (Constitution IV: *fix the trait gap in its own
corpus-gated PR first, then land the dependent change*). B3.3a.2 is that follow-up trait
gap; **B3.3b stays engine-only** and consumes all seven accessors (four from B3.3a, three
from here).

## What lands (additive; engine behavior byte-identical)

### marque-scheme — `MarkingScheme`
- **`canonical_from_marking(&self, &Self::Marking) -> Self::Canonical`** — inverse of
  `marking_from_canonical`. Default `unimplemented!()` (mirrors `canonicalize` /
  `marking_from_canonical`). CapcoScheme → `marking.0.clone()` (drops the provenance
  side-channel at tuple-position 1).
- **`canonical_rank(&self, &Self::Canonical) -> Option<u8>`** — the monotone sensitivity
  rank used as the per-page recognition floor. Default `None` (no rank axis — additive).
  CapcoScheme → `classification.as_ref().map(|c| c.effective_level() as u8)`.

Both bounds are on the methods / inherited from the default; no associated-type bound is
tightened (the lift stays purely additive, per the B3.3a R2 lesson).

### marque-rules — `ConstraintBridge` + `RecognitionOutcome<S>`
- **`RecognitionOutcome<S> { is_decoder_path: bool, recognition_score: Option<f32>,
  diagnostic: Option<Diagnostic<S>> }`** — what the engine learns from a recognized
  marking beyond the marking itself.
- **`recognition_outcome(&self, &Self::Marking, span, original_bytes, kind,
  corpus_override_active) -> RecognitionOutcome<Self>`** — default returns the strict-path
  outcome (`false` / `None` / `None`), correct for any scheme with a purely strict
  recognizer. CapcoScheme reads `marking.1` and delegates diagnostic synthesis to
  `build_decoder_diagnostic`.

This hook lives on `ConstraintBridge` (marque-rules), not `MarkingScheme`, for the same
dependency-direction reason as the other bridge methods: it returns `Diagnostic<Self>`,
and `marque-scheme` is the graph leaf.

### marque-capco — relocation (forced) + impls
`build_decoder_diagnostic` and its constants (`DECODER_RULE_ID`, `DECODER_CITATION_TYPED`,
`HEURISTIC_RECOGNITION_CAP`) **move from `marque-engine` (`engine/synthesis.rs`,
`engine.rs`) into `marque-capco` (`provenance.rs`)**. This is forced, not stylistic: the
CapcoScheme `recognition_outcome` impl must synthesize a `Diagnostic<CapcoScheme>` from
`DecoderProvenance` (a marque-capco type), and capco cannot call an engine function
(engine → capco only). Exposing `DecoderProvenance` generically would require a
source-breaking associated type; relocation is the only additive path. The `"engine"`
scheme string on `DECODER_RULE_ID` is unchanged, so the audit wire-form is byte-identical
(capco now *constructs* the engine-namespaced sentinel — semantically fine; the namespace
is a string, not a crate boundary). `secrecy` is promoted from a capco dev-dependency to a
normal dependency (`build_decoder_diagnostic` wraps the canonical bytes in `SecretBox`);
it is WASM-safe (marque-rules already normal-deps it for the same field).

The engine's single call site (`lint_helpers::recognize_marking_candidate`) keeps calling
`build_decoder_diagnostic` directly (now imported from `marque_capco`), reading `marking.1`
itself — the engine stays `CapcoScheme`-pinned in B3.3a.2. B3.3b swaps that call site for
`scheme.recognition_outcome(...)`.
