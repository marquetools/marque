# Phase B3.3b.2 — fix-pipeline generification (engine-only)

## Position in the split

B3.3b generified the **lint** pipeline to `impl<S: MarkingScheme +
ConstraintBridge, R: Recognizer<S>> Engine<S, R>`. B3.3b.2 completes the engine
long pole by generifying the **fix** pipeline, which until now stayed
`Engine<CapcoScheme, EngineRecognizer>`-pinned and called the generic lint at
`S = CapcoScheme`.

The two halves were severable precisely because `lint_with_options_internal*`
already returns `Vec<(Span, S::Marking)>`; the pinned fix path consumed that at
`S = CapcoScheme`. This PR removes the pin.

## No trait-lift needed — pure type-threading

Unlike B3.3a / B3.3a.2 (which had to *lift* CAPCO-inherent methods onto the
trait surface before the lint pipeline could reach them generically), every
scheme method the fix path calls is **already on the `MarkingScheme` trait**:

| Fix-path call | Trait method (already present) |
|---------------|--------------------------------|
| `scheme.apply_intent(marking, &intents)` | `MarkingScheme::apply_intent` |
| `scheme.render_item(&modified)` | `MarkingScheme::render_item` |
| `scheme.render_summary(&modified)` | `MarkingScheme::render_summary` |
| `scheme.category_of(token)` | `MarkingScheme::category_of` |
| `marking.0.clone()` (the canonical) | `MarkingScheme::canonical_from_marking` (B3.3a.2) |

So B3.3b.2 is mechanical type-threading + two output-type generifications —
the same shape as B3.3b (lint), not a trait-lift.

## What lands (engine-only; behavior byte-identical at `S = CapcoScheme`)

| Site | Change |
|------|--------|
| `output.rs` `FixResult` | `FixResult<S = CapcoScheme>` holding `Vec<AuditLine<S>>` + `Vec<Diagnostic<S>>`; derived `Debug` (the `S: Debug` bound lands only on the `Debug` impl, never the struct — same as `Diagnostic<S>` / `LintResult<S>`); `impl<S: MarkingScheme> FixResult<S>` accessors return `&AppliedFix<S>` |
| `text_correction.rs` `SynthesizedFix` | `SynthesizedFix<S = CapcoScheme>` with `intent: FixIntent<S>`; derived `Debug`, hand-written `Clone` bounded on `S: MarkingScheme` (a `Clone` derive would impose a spurious `S: Clone`; mirrors `FixIntent<S>`'s own hand-written `Clone`) |
| `synthesis.rs` free fns | thread `<S: MarkingScheme>` through `find_containing_marking`, `lookup_marking` (`-> Option<&S::Marking>`), `sort_and_c1_dedup`, `splice_fixes_forward`, `synthesize_fixes`, `apply_fr023_and_i18`, `build_r002_diagnostic` (`-> Diagnostic<S>`) |
| `fix.rs` impl block + `TwoPassFixer` | `impl<S, R> Engine<S, R> where S: MarkingScheme + ConstraintBridge, S::Canonical: Clone + Default + PartialEq, R: Recognizer<S>`; `TwoPassFixer<'engine, S, R>` holds `&Engine<S, R>`; public `fix` / `fix_with_threshold` / `fix_with_options` return `FixResult<S>` |
| `fix_impl.rs` pass results + run methods | `Pass0Result<S>` / `Pass1Result<S>` / `Pass2Result<S>`; `run*` methods generic; `populate_pre_pass_1_cache` swaps `marking.0.clone()` → `scheme.canonical_from_marking(marking)` |

Type swaps throughout: `CapcoScheme → S`, `CapcoMarking → S::Marking`,
`CanonicalAttrs → S::Canonical`, `Diagnostic<CapcoScheme> → Diagnostic<S>`,
`AuditLine<CapcoScheme> → AuditLine<S>`, `FixIntent<CapcoScheme> → FixIntent<S>`,
`ReplacementIntent<CapcoScheme> → ReplacementIntent<S>`,
`EngineConstructor::<CapcoScheme> → ::<S>`, `Canonical<CapcoScheme> → Canonical<S>`,
`AppliedFix::<CapcoScheme>::__engine_promote → ::<S>::`.

## Bounds — identical to the lint block (because fix re-lints)

`TwoPassFixer::run` calls `self.engine.lint_with_options_internal_with_source`,
which lives on the lint block. So the fix impl carries the **same** bound set:
`S: MarkingScheme + ConstraintBridge`, `S::Canonical: Clone + Default +
PartialEq`, `R: Recognizer<S>`. No new bound is introduced.

`PrePass1Cache` becomes `PrePass1Cache<S> = SmallVec<[(Span, S::Canonical); 4]>`
so `&pre_pass_1_cache` deref-coerces to the `Option<&[(Span, S::Canonical)]>`
the lint method already accepts (B3.3b).

## What stays concrete (deliberately)

- **`TextCorrectionProposal`** — a `(span, replacement, confidence, source,
  message, migration_ref)` carrier with no `S`-typed field. Text corrections are
  pure byte substitutions of corpus-derived canonical tokens; the audit record
  (`AppliedTextCorrection`, also non-generic in `marque-rules`) carries no
  scheme payload. Both stay concrete.
- **`dispatch.rs`** helpers (`partition_rules_by_phase`, `build_severity_tables`,
  `canonicalize_rule_overrides`) — reached only from `Engine::new`, which stays
  `CapcoScheme`-pinned until B3.4. Their `&[Box<dyn RuleSet<CapcoScheme>>]`
  parameters move when the constructors generify, not here.
- **The decoder / `EngineRecognizer`** — the concrete `Recognizer<CapcoScheme>`,
  reached generically via `R` (B3.2). Unchanged.

## `__engine_promote` generic-readiness

`AppliedFix<S>::__engine_promote` and `AppliedTextCorrection::
__engine_promote_text_correction` are **already** generic / scheme-neutral in
`marque-rules`. B3.3b.2 only changes the call site from `::<CapcoScheme>::` to
`::<S>::`. The engine-only contract (Constitution V) is unchanged: the sole
production call site stays `Engine::fix_inner`'s pass machinery
(`apply_kept_fixes` / `apply_text_corrections`), now generic over `S`.

## Reachability test

A compile-time monomorphization guard at
`crates/engine/tests/b3_3b2_generic_fix_reachability.rs` (mirroring
`b3_3b_generic_lint_reachability.rs`): a `const` function pointer pinning the
generic fix surface at `Engine<StubScheme, StubRecognizer>` forces the compiler
to monomorphize the fix path for a non-CAPCO scheme. A live run stays deferred
until the constructors generify (B3.4) — `Engine<StubScheme>` is not yet
constructible.

## Recurring-lesson controls (carried from B3.3b)

1. **Doc-accuracy sweep:** when rewriting one doc comment for generification,
   grep all sibling docs in the same file for the same concrete-type references
   in the same pass (the defect class that recurred at every generification PR).
2. **Hardcoded-literal sweep:** grep the bodies for scheme-id string literals
   (`"capco"`, predicate prefixes) — not just type names. The StubScheme
   reachability test compiles a wrong-runtime-value leak fine; only a
   second-scheme behavioral test (deferred to B3.4) catches it, so the literal
   grep is the compensating control until then.
