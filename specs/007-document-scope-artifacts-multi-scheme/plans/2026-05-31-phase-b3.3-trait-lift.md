# Phase B3.3 — canonical-space + constraint-bridge trait lift, then engine generification

## Why this is split

B3.1 (`RuleContext<'a, S>` + `Rule::check` via `MarkingScheme::Projected`, #837) and
B3.2 (generic recognizer param `R`, #838) were achievable engine-only. B3.3 — the
pipeline-generification long pole — is **not**. The lint/fix pipeline operates
predominantly in **canonical space** and calls CAPCO-**inherent** methods that are
absent from the `MarkingScheme` trait:

| Inherent call (today)                                   | Space           | Defined in |
|---------------------------------------------------------|-----------------|------------|
| `CapcoMarking::join_via_lattice(&[CanonicalAttrs])`     | Canonical→Canonical | `capco/src/scheme/marking.rs` |
| `CapcoScheme::project_from_attrs_slice[_with_sink]`     | Canonical→Canonical | `capco/src/scheme/marking_scheme_impl.rs` |
| `From<CanonicalAttrs> for CapcoMarking`                 | Canonical→Marking | `capco/src/scheme/marking.rs` |
| `CapcoScheme::has_diagnostic_constraints`               | —               | `capco/src/scheme/adapter.rs` |
| `CapcoScheme::fix_intent_by_name`                       | →`FixIntent<S>` | `capco/src/scheme/adapter.rs` |
| `CapcoScheme::message_by_name`                          | →`Message`      | `capco/src/scheme/adapter.rs` |
| `CapcoScheme::bridge_sci_per_system_diagnostics`        | →`Vec<Diagnostic<S>>` | `capco/src/scheme/adapter.rs` |

The `MarkingScheme` trait is defined in **marking space** (`project(&[Self::Marking])`,
`validate(&Self::Marking)`); the engine accumulates pages in `Self::Canonical`. That
space mismatch is the long pole.

**Dependency-direction constraint (decisive for placement):** `FixIntent`, `Message`,
and `Diagnostic` live in `marque-rules`, and `marque-scheme` is the graph leaf — it
cannot depend on `marque-rules`. Therefore the four constraint-bridge methods **cannot**
sit on the `MarkingScheme` trait. They go on a new trait in `marque-rules` (which already
depends on both `marque-scheme` and `marque-ism`, and already names `S::Canonical` /
`Diagnostic<S>` in `Rule::check`). Only the canonical-space ops sit on `MarkingScheme`.

## B3.3a — trait lift (marque-scheme + marque-rules + marque-capco). Engine untouched.

Behavior-identical: every new trait method's CapcoScheme body **delegates to the existing
inherent body verbatim**. The engine keeps calling the inherent methods until B3.3b, so
the full corpus / audit-parity / G13 / perf gates must stay byte-identical.

### marque-scheme `MarkingScheme`

1. **Add bounds** to the canonical associated type:
   `type Canonical: Clone + Default + core::fmt::Debug + Send + Sync + 'static;`
   The generic engine clones it, `Default`-seeds the page accumulator, `debug_assert`s on
   it, and `BatchEngine` threads it across workers (Constitution VI). `CanonicalAttrs`
   already satisfies all five; the stub `type Canonical = ()` satisfies all five.

2. **`fn canonical_page_join(&self, portions: &[Self::Canonical]) -> Self::Canonical`** —
   default body `portions.last().cloned().unwrap_or_default()` (now expressible via the new
   bounds). CapcoScheme overrides → `CapcoMarking::join_via_lattice(portions)`.

3. **`fn project_canonical(&self, portions: &[Self::Canonical]) -> Self::Projected`** (the
   non-instrumented hot path) and its sink-aware companion
   **`fn project_canonical_with_sink(&self, portions, sink: &mut dyn DecisionSink) -> Self::Projected`** —
   both default `unimplemented!()` (mirrors `canonicalize`; stub schemes never call them).
   CapcoScheme overrides → `ProjectedMarking::from_canonical(self.project_from_attrs_slice(portions))`
   and (`#[cfg(feature = "decision-tracing")]`, mirroring `project_with_sink`)
   `ProjectedMarking::from_canonical(self.project_attrs_pipeline_with_sink(portions, sink))`. Two
   methods (not one) so the non-tracing hot path stays on `project_from_attrs_slice` rather than
   routing through a no-op sink.

4. **`fn marking_from_canonical(&self, canonical: Self::Canonical) -> Self::Marking`** —
   default `unimplemented!()` (mirrors `canonicalize`). CapcoScheme overrides →
   `CapcoMarking::from(canonical)`.

### marque-rules — new `ConstraintBridge` trait

```rust
pub trait ConstraintBridge: MarkingScheme + Sized {
    fn has_diagnostic_constraints(&self) -> bool { false }
    fn fix_intent_by_name(
        &self, name: &str, canonical: &Self::Canonical, kind: marque_ism::MarkingType,
    ) -> Option<FixIntent<Self>> { let _ = (name, canonical, kind); None }
    fn message_by_name(
        &self, name: &str, canonical: &Self::Canonical, kind: marque_ism::MarkingType,
    ) -> Option<Message> { let _ = (name, canonical, kind); None }
    fn bridge_sci_per_system_diagnostics(
        &self, canonical: &Self::Canonical, span: Span, scope: Scope,
        overrides: &SeverityOverrides,
    ) -> Vec<Diagnostic<Self>> { /* … */ Vec::new() }
}
```

All four default to the "no constraints" behavior so a future scheme that declares no
diagnostic constraints gets the right empty behavior for free. CapcoScheme's impl (in
marque-capco) delegates to today's inherent bodies verbatim. Confirm the exact
`bridge_sci_per_system_diagnostics` override-table parameter type against
`adapter.rs:813` during impl (it takes the engine's `emitted_id_overrides` map).

### marque-capco

- `impl ConstraintBridge for CapcoScheme` delegating to the inherent methods.
- The four `MarkingScheme` overrides delegating to the inherent canonical-space bodies.
- The inherent methods stay (still called by the engine until B3.3b); no logic moves.
- Tests: exercise each new trait method through `CapcoScheme` and assert identical output
  to the inherent method for a representative fixture set; confirm the stub scheme inherits
  the no-op `ConstraintBridge` defaults.

### B3.3a gates

`fmt`; `cargo check --workspace --all-targets` (1.89); `clippy -D warnings` (stable) on
each touched crate; full `cargo test`; **corpus ≥95% lint+fix**; audit-parity; G13 canary;
`wasm-pack build crates/wasm` (all three touched crates are WASM-safe — verify nothing
leaked); interactive-latency bench (no engine change ⇒ expect flat).

## B3.3b — engine pipeline generification (marque-engine)

With the trait surface in place: lift the lint/fix pipeline to
`impl<S: MarkingScheme + ConstraintBridge> Engine<S, R>` where `R: Recognizer<S>`; replace
the ~116 `CanonicalAttrs` → `S::Canonical` and ~88 `CapcoMarking` → `S::Marking`; swap the
inherent call-sites to the trait methods; remove the B2 decoder shim. Same gate set, with
the interactive-latency bench as a hard gate (Constitution I) and the static
`assert_impl_all!(Engine: Send, Sync)`. Reviewer chain (rust-reviewer + code-reviewer)
before PR-open. After B3.3b, the now-unused inherent methods on `CapcoScheme` either become
the trait-impl bodies directly or are removed if fully subsumed.

## Sequencing note

This mirrors Constitution IV's "fix the trait gap in its own corpus-gated PR first, then
land the dependent change" discipline. B3.3a is purely additive (new trait methods, unused
by the engine), so its blast radius is the three WASM-safe crates and its risk is contained
by behavior-identity. B3.3b carries the actual generification and the perf hard-gate.
