# Phase B3.4 — close the constructor scheme-discard

## Position in the split

B3.1–B3.3b.2 generified the engine's **method blocks** (`RuleContext<S>`,
generic recognizer `R`, canonical-space trait-lift, lint pipeline, fix
pipeline) to `impl<S, R> Engine<S, R>`. But the **constructors** stayed pinned
to `impl Engine<CapcoScheme, EngineRecognizer>`: `Engine::new` / `with_clock`
accept a generic `scheme: S` used only to extract `page_rewrites()` for the
scheduler, then `drop(scheme)` it and store a fresh `CapcoScheme::new()`.

So `Engine<S>` is *typeable* for a second scheme but not *constructible* — every
construction path lands on `CapcoScheme`. B3.4 closes that discard.

## Why the discard exists (the B2 vestige)

Two reasons, both now resolvable:

1. **Recognizer pinning.** The constructors hard-wire
   `EngineRecognizer::default()`, which is `Recognizer<CapcoScheme>` only. A
   generic constructor that stores the scheme must also accept the recognizer
   explicitly (it can't synthesize a `Recognizer<S>` for an arbitrary `S`).
2. **One inherent-method blocker.** `canonicalize_rule_overrides` /
   `build_severity_tables` call `scheme.bridge_emitted_rule_ids()`, which is an
   **inherent** method on `CapcoScheme` (`adapter.rs`), not on any trait. A
   generic `&S` can't reach it.

`validate_intent_rewrites<S>` and `schedule_rewrites<S>` are **already** generic
over `S: MarkingScheme` — no change needed there.

## What lands

| Site | Change |
|------|--------|
| `crates/rules/src/constraint_bridge.rs` | Add `bridge_emitted_rule_ids(&self) -> &'static [(&'static str, &'static str)]` to `ConstraintBridge` with a `&[]` default. The natural home: it feeds the same bridge-diagnostic config-key surface the trait already governs. |
| `crates/capco/src/scheme/adapter.rs` | Move CapcoScheme's inherent `bridge_emitted_rule_ids` body into its `impl ConstraintBridge for CapcoScheme` block. `StubScheme` inherits the `&[]` default (empty bridge). |
| `crates/engine/src/engine/dispatch.rs` | `partition_rules_by_phase<S: MarkingScheme>`, `build_severity_tables<S: MarkingScheme>`, `canonicalize_rule_overrides<S: ConstraintBridge>` — `&[Box<dyn RuleSet<CapcoScheme>>]` → `&[Box<dyn RuleSet<S>>]`, `&CapcoScheme` → `&S`. |
| `crates/engine/src/engine/constructors.rs` | New generic core `impl<S, R> Engine<S, R> where S: MarkingScheme + ConstraintBridge, R: Recognizer<S>` with `with_clock_and_recognizer(config, rule_sets: Vec<Box<dyn RuleSet<S>>>, scheme: S, recognizer: R, clock)` that stores the passed scheme + recognizer — **no `drop`, no `CapcoScheme::new()`**. CAPCO `new`/`with_clock` become concrete wrappers (`scheme: CapcoScheme`) delegating with `EngineRecognizer::default()`. |
| `crates/engine/src/engine.rs` | Rewrite the `scheme` field doc: drop the "always stores a fresh `CapcoScheme::new()` … discard closes in PR-B3" prose; the field now stores the user-supplied `S`. |

## Constructor shape (the no-discard core)

```rust
// Generic core — stores both the scheme and the recognizer.
impl<S, R> Engine<S, R>
where
    S: MarkingScheme + ConstraintBridge,
    R: Recognizer<S>,
{
    pub fn with_clock_and_recognizer(
        mut config: Config,
        rule_sets: Vec<Box<dyn RuleSet<S>>>,
        scheme: S,
        recognizer: R,
        clock: Box<dyn Clock>,
    ) -> Result<Self, EngineConstructionError> {
        canonicalize_rule_overrides(&mut config, &rule_sets, &scheme)?;
        validate_intent_rewrites(&scheme, scheme.page_rewrites())?;
        let scheduled_rewrites = schedule_rewrites(scheme.page_rewrites())?;
        // … corrections-map / phase-partition / severity-table prep …
        Ok(Self { config, rule_sets, scheme, recognizer, /* … */ })
    }
}

// CAPCO convenience — concrete, source-unchanged for every existing caller.
impl Engine<CapcoScheme, EngineRecognizer> {
    pub fn new(
        config: Config,
        rule_sets: Vec<Box<dyn RuleSet<CapcoScheme>>>,
        scheme: CapcoScheme,
    ) -> Result<Self, EngineConstructionError> {
        Self::with_clock(config, rule_sets, scheme, Box::new(SystemClock))
    }
    pub fn with_clock(
        config: Config,
        rule_sets: Vec<Box<dyn RuleSet<CapcoScheme>>>,
        scheme: CapcoScheme,
        clock: Box<dyn Clock>,
    ) -> Result<Self, EngineConstructionError> {
        Self::with_clock_and_recognizer(config, rule_sets, scheme,
            EngineRecognizer::default(), clock)
    }
}
```

The heavy construction body is now generic, so it monomorphizes per
*(scheme, recognizer)* pair actually in use — one instantiation for the CAPCO
path, plus the test stub. This supersedes the old `with_clock_prepared`
concrete-tail split, whose only purpose was to avoid monomorphizing the body
for the throwaway `S` values passed to `new<S>`; those values are no longer
throwaway.

## Source-compat

Every production caller passes `default_scheme()` (a `CapcoScheme`) and
`Vec<Box<dyn RuleSet<CapcoScheme>>>` to the 3-arg `Engine::new` — unchanged. The
only non-CAPCO caller of `new<S>` today is `crates/engine/tests/scheduler.rs`,
which passed a local stub scheme purely to drive the scheduler. It migrates to
the generic constructor (below).

## Tests

1. **`scheduler.rs` migration.** Its local rewrite-configurable `StubScheme`
   moves off `Engine::new` onto `with_clock_and_recognizer`, returning a true
   `Engine<StubScheme, StubRecognizer>`. Add `impl ConstraintBridge for
   StubScheme {}` (empty) and a local `StubRecognizer`. `scheduled_rewrites()`
   reads through the existing generic accessor.
2. **Live second-scheme lint+fix.** The B3.3b / B3.3b.2 reachability guards were
   compile-time-only (`Engine<StubScheme>` was not constructible). Now upgrade to
   a live run: construct `Engine<StubScheme, StubRecognizer>` via the generic
   constructor, run `lint` + `fix` on sample input, assert empty/clean results
   (the stub recognizes nothing, declares no constraints/rules), **and** assert
   `engine.scheme().scheme_id() == "stub"`. That last assertion is the direct
   leak-catcher: under the old discard `engine.scheme()` returned the
   substituted `CapcoScheme::new()` (id `"capco"`); a hardcoded-`"capco"` leak a
   compile-time const-fn-ptr guard cannot see fails here at runtime.
