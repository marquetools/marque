# Phase B3.3b — lint-pipeline generification (engine-only)

## Why a split

B3.3b (the engine long pole) generifies the lint/fix pipeline from
`impl Engine<CapcoScheme, R>` to `impl<S: MarkingScheme + ConstraintBridge>
Engine<S, R>`. Recon (2026-06-01) measured 314 `CapcoScheme` / 116 `CanonicalAttrs`
/ 88 `CapcoMarking` references across 40 engine `src` files. The lint and fix
halves generify **independently**: `lint_with_options_internal*` returns
`Vec<(Span, S::Marking)>`, and a still-`CapcoScheme`-pinned fix path calls the
now-generic lint at `S = CapcoScheme` and receives exactly the
`Vec<(Span, CapcoMarking)>` it consumes today. So:

- **B3.3b (this PR):** the lint pipeline.
- **B3.3b.2 (next):** the fix pipeline (`fix.rs`, `fix_impl.rs`, `synthesis.rs`,
  `TwoPassFixer`, `AppliedFix<S>` promotion).

## Key recon insight — the recognizer/decoder stays pinned

The decoder (`crates/engine/src/decoder/`) and `EngineRecognizer`
(`recognizer.rs`) legitimately **stay `CapcoScheme`-bound** — they *are* the
concrete `Recognizer<CapcoScheme>` implementation, reached generically through
the Fork-B `R: Recognizer<S>` type parameter (B3.2). The generification target
is the **pipeline** (`crates/engine/src/engine/`), not the recognizer. This is
what bounds the work: the ~88 `CapcoMarking` refs in `decoder/` are inside the
concrete recognizer and do not move.

## `RecognizedCandidate` design — Option A (hold marking whole)

`recognize_marking_candidate` currently destructures the recognizer output:
`let attrs = marking.0` (move) + `provenance: marking.1` (move). Generically the
output is an opaque `S::Marking`; the engine can only borrow it
(`canonical_from_marking(&m)`, `recognition_outcome(&m, ..)`) — it cannot move
the canonical out without a consuming accessor the trait does not expose.

**Chosen shape:** `RecognizedCandidate<S> { marking: S::Marking, attrs:
S::Canonical, diagnostics_pre_candidate }`.

- `attrs = canonical_from_marking(&marking)` — recognize needs an owned canonical
  for `canonical_rank` anyway; it hands the same clone to the pipeline for the
  page accumulator.
- The whole `marking` is pushed into `parsed_markings` (the lint→fix cache),
  **preserving provenance** — exactly byte-identical to today's
  `CapcoMarking(attrs, provenance)` reconstruction.

Provenance was independently verified **dead downstream**: the only fix-path read
of a `parsed_markings` marking is `fix_impl.rs::populate_pre_pass_1_cache` doing
`marking.0.clone()` (the canonical); `.1` (provenance) is never consumed. So even
the provenance-dropping `marking_from_canonical` reconstruction would be
behavior-safe — but Option A preserves it for zero semantic change.

**Perf note (the one cost).** In the common no-fix portion path, Option A adds
**one** `S::Canonical` clone per recognized portion versus today's free
move-out (`attrs = marking.0`). For CapcoScheme a typical portion's
`CanonicalAttrs` is mostly empty `Box<[T]>` fields, so the clone is cheap; with
~4× interactive-latency headroom (≈430–480 µs against the 2 ms SC-001 ceiling)
this is expected to be in the noise. **The interactive-latency bench is run
before/after** (Invariant 5 of the parent plan). If it regresses past tolerance,
the escalation is an `into_canonical(S::Marking) -> S::Canonical` consuming
accessor via a B3.3a.3 addendum (trait gap first, then engine) — not a hack in
the engine.

## Bounds placement

All bounds live on the **engine impl block**, never on the `MarkingScheme` trait
(the B3.3a R2 lesson: tightening an associated-type bound is source-breaking).
The minimal set the compiler requires is `S::Canonical: Clone + Default +
PartialEq`: `Clone` for the page accumulator (`recognized.attrs.clone()` into the
join + `to_vec` snapshot), `Default` for the `page_join_acc` reset and the
`dispatch_page_finalization` dummy attrs, `PartialEq` for the debug-only
page-finalization read-only sentinel (`check_portions_unchanged`'s `before ==
after`, satisfied universally). No `Debug` / `Send` / `Sync` / `'static` bound is
needed — the canonical is never engine-Debug-formatted (the G13 sentinel avoids
`Debug` deliberately) nor sent across threads within a lint pass.
`S::Projected` already carries `Debug + Clone + Send + Sync + 'static` on the
trait, so no extra impl bound is needed for it.

`LintResult<S = CapcoScheme>` carries `Vec<Diagnostic<S>>`, so generifying the
lint pipeline requires generifying its output type (the natural completion of
`Diagnostic<S>`). The `#[derive(Debug)]` is kept (consistent with `Diagnostic<S>`,
which also derives `Debug` and so needs `S: Debug` when formatted), but `Default`
and `Clone` are **hand-written** bounded only on `S: MarkingScheme` — a derive
would impose a spurious `S: Default` / `S: Clone` that the generic pipeline's
`..Default::default()` does not have (same rationale as `Diagnostic<S>` /
`RuleContext<'_, S>`). The `= CapcoScheme` default keeps every CLI / server / WASM
/ fix-path caller that names bare `LintResult` source-unchanged. `FixResult`
stays concrete (`Vec<Diagnostic<CapcoScheme>>`) — the fix path is B3.3b.2.

The decision-tracing and constructor accessor blocks generify to the looser
`impl<S: MarkingScheme, R>` (no `ConstraintBridge`) since they never touch
scheme-specific behavior; the `ConstraintBridge`-bounded lint block can still
call them because `ConstraintBridge: MarkingScheme`.

`fix.rs` / `fix_impl.rs` / `synthesis.rs` stay `Engine<CapcoScheme>`-pinned; they
call the now-generic lint at `S = CapcoScheme` and consume the concrete
`Vec<(Span, CapcoMarking)>` unchanged. That is B3.3b.2.
