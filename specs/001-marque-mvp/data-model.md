# Phase 1 Data Model: Marque MVP

**Branch**: `001-marque-mvp` | **Date**: 2026-04-08

The canonical types for the MVP slice. Implemented in `marque-ism` (ISM
vocabulary types and generated CVE enums), `marque-core` (scanner and parser),
and `marque-rules` (trait surface). Hand-written Layer 2 rules in `marque-capco`
consume these types.

All types are `#[non_exhaustive]` where they cross a crate boundary, so future
slices can add fields without a breaking change.

---

## Span

```rust
#[derive(Copy, Clone, Eq, PartialEq, Hash, Debug)]
pub struct Span {
    pub start: usize, // inclusive byte offset
    pub end: usize,   // exclusive byte offset
}
```

**Invariants**: `start <= end`; both indices are byte offsets into the original
input buffer; never copies content. The original buffer must outlive any `Span`
that references it.

**Source**: spec key entity *Marking Candidate*; constitution Principle II.

## IsmAttributes

```rust
#[non_exhaustive]
#[derive(Clone, Eq, PartialEq, Debug)]
pub struct IsmAttributes {
    pub classification: Option<Classification>,
    pub sci_controls: Box<[SciControl]>,
    pub sar_identifiers: Box<[SarIdentifier]>,
    pub dissem_controls: Box<[DissemControl]>,
    pub rel_to: Box<[Trigraph]>,
    pub declassify_on: Option<DeclassDate>,
    pub classified_by: Option<Box<str>>,
    pub derived_from: Option<Box<str>>,
    pub declass_exemption: Option<DeclassExemption>,
}
```

**Field types** (`Classification`, `SciControl`, `SarIdentifier`,
`DissemControl`, `Trigraph`, `DeclassDate`, `DeclassExemption`) are
code-generated by `marque-ism/build.rs` from the ODNI CVE XML enumerations
under `crates/marque-ism/schemas/ISM-v2022-DEC/CVE_ISM/`. Each is a closed enum
derived from a single CVE file (R-1). Both `IsmAttributes` and its field types
live in `marque-ism`, resolving the circular dependency that would otherwise
exist between `marque-core` (consumer) and `marque-capco` (generator).

**Allocation**: every variable-length field uses `Box<[T]>`, never `Vec<T>`,
per constitution Principle II. Optional strings use `Box<str>`.

**Validation**: `IsmAttributes` itself enforces only structural well-formedness.
Semantic validity (allowed combinations of dissem controls, REL TO ordering,
etc.) is the job of the Layer 1 generated predicates and the Layer 2 rule set.

**Source**: spec key entity *ISM Attributes*; design doc §7.

## Severity

```rust
#[derive(Copy, Clone, Eq, PartialEq, Ord, PartialOrd, Hash, Debug)]
pub enum Severity { Off, Warn, Error, Fix }
```

**Source**: design doc §8 config model. `Off` disables the rule entirely; `Fix`
means "report and apply if confidence ≥ threshold". `Severity` is `Ord` so
config merging can use `max` semantics if multiple layers set the same rule.

## FixProposal

```rust
#[non_exhaustive]
#[derive(Clone, Debug)]
pub struct FixProposal {
    pub rule: RuleId,
    pub source: FixSource,       // provenance; see below
    pub span: Span,              // the range being replaced (mirrors parent Diagnostic.span)
    pub original: Box<str>,      // the bytes currently occupying `span`
    pub replacement: Box<str>,
    pub confidence: f32,         // [0.0, 1.0], NaN forbidden
    pub migration_ref: Option<&'static str>,
}

#[derive(Copy, Clone, Eq, PartialEq, Hash, Debug)]
pub enum FixSource {
    BuiltinRule,      // hand-written Layer 2 CAPCO rule
    CorrectionsMap,   // user `[corrections]` entry (FR-009)
    MigrationTable,   // deterministic deprecated-marking conversion (FR-004a)
}
```

**Invariants**: `0.0 <= confidence <= 1.0`, `!confidence.is_nan()`. A
`FixProposal` is pure data — deterministic, timestamp-free, classifier-free,
safe to snapshot in tests.

**Lifecycle**: a `FixProposal` is attached to a `Diagnostic` by a rule. It is a
*suggestion* until `Engine::fix` runs; at that point, proposals with
`confidence >= configuration.confidence_threshold` are promoted to
`AppliedFix` (below). Sub-threshold proposals remain suggestions in the
diagnostic stream and never produce an `AppliedFix`.

**Source**: spec key entity *Fix*; FR-004, FR-004a. Supersedes the original
single-type `Fix` struct: splitting proposal from application makes the
"suggested vs applied" distinction a type-system invariant rather than a
runtime flag, and removes the timestamp from suggestions that never happened.

## AppliedFix

```rust
#[non_exhaustive]
#[derive(Clone, Debug)]
pub struct AppliedFix {
    pub proposal: FixProposal,
    pub timestamp: SystemTime,       // clock-injected; see Clock seam below
    pub classifier_id: Option<Box<str>>,
    pub dry_run: bool,               // true if produced under --dry-run (FR-006)
    pub input: Option<Box<str>>,     // caller-supplied input identifier (file path, "-" for stdin, None if not applicable)
}

**Total fields invariant**: `AppliedFix = FixProposal + {timestamp,
classifier_id, dry_run, input}`. No other fields. The struct is
`#[non_exhaustive]` to permit future additions without a breaking change,
but new fields require a bump to the audit-record contract version
(`marque-mvp-2`). A contract test (T045) round-trips `AppliedFix → NDJSON
→ schema validate` to keep the Rust type and `contracts/audit-record.json`
in lock-step.

**`input` field**: populated by the CLI (or other embedder) at the boundary
just before audit emission — rules and the engine core never set it.
For `marque fix file.txt` the value is `Some("file.txt")`; for piped
stdin it is `Some("-")`; for library callers that pass no identifier it
is `None`. This supports the Compliance Reviewer stakeholder's "what
was changed, in which file" question without coupling the engine to the
filesystem.
```

**Invariants**: immutable on construction. Constructed **only** by
`Engine::fix` at the moment a `FixProposal` meets the threshold — never by a
rule, never by a suggestion path. `classifier_id` is `None` unless the runtime
configuration supplied one via `MARQUE_CLASSIFIER_ID` or `.marque.local.toml`;
committed config files MUST NOT supply it (FR-010, SC-006, R-5).

**`dry_run: bool`**: under `--dry-run` (FR-006), `AppliedFix` records are
constructed identically to a real run but no bytes are written to disk. The
audit stream is byte-identical between a dry-run and a real run modulo this
flag, satisfying the spec's US2 acceptance scenario "`--dry-run` produces the
full audit output without modifying the file."

**Clock seam**: `timestamp` is produced by a `Clock` trait injected into
`Engine` (defaulting to `SystemTime::now()`). Tests inject a fixed clock so
snapshot tests of the audit stream are deterministic.

**`AuditRecord` = `AppliedFix`**: the spec's key entity *Audit Record* is now
materialized as `AppliedFix`. The NDJSON schema at `contracts/audit-record.json`
(schema version `marque-mvp-1`) serializes `AppliedFix`, not `FixProposal`;
suggestions never appear in the audit stream.

**Source**: spec key entity *Audit Record*; FR-005, FR-006, FR-010, SC-004,
SC-006.

## Configuration

```rust
#[non_exhaustive]
#[derive(Clone, Debug)]
pub struct Configuration {
    pub schema_version: SchemaVersion, // pinned in marque-ism; see FR-011
    pub rule_severities: BTreeMap<RuleId, Severity>,
    pub corrections: PhfMap<&'static str, &'static str>, // built from [corrections]
    pub confidence_threshold: f32, // default 0.95
    pub classifier_id: Option<Box<str>>, // never from committed config
    pub classification_authority: Option<Box<str>>,
}
```

**Construction**: `marque-config::Loader::load()` returns a `Configuration`
built by merging four layers in order of increasing precedence:

1. `.marque.toml` (committed) — provides `rule_severities`, `corrections`,
   `schema_version`. May NOT provide `classifier_id`.
2. `.marque.local.toml` (gitignored) — provides `classifier_id`,
   `classification_authority`.
3. Environment variables (`MARQUE_CLASSIFIER_ID`, `MARQUE_LOG`,
   `MARQUE_CONFIDENCE_THRESHOLD`, …).
4. CLI flags (`--confidence-threshold`, `--config`, …).

**Validation rules**:
- `0.0 <= confidence_threshold <= 1.0`
- `schema_version` must match the value pinned in
  `marque-ism/Cargo.toml [package.metadata.marque] ism-schema-version`. A
  mismatch is a fatal load error.
- A `[user]` section in `.marque.toml` is a fatal load error.

**Schema version pinning — concrete contract**: `crates/marque-ism/Cargo.toml`
carries exactly this block:

```toml
[package.metadata.marque]
ism-schema-version = "ISM-v2022-DEC"
```

`marque-ism/build.rs` reads this metadata via `CARGO_PKG_METADATA_*` or by
re-parsing `Cargo.toml` with `toml`, asserts the value matches the schema
directory name under `schemas/`, and emits
`pub const SCHEMA_VERSION: &str = "ISM-v2022-DEC";` into
`OUT_DIR/values.rs` (consumed via `src/generated.rs`). A build-time mismatch (schema dir renamed,
metadata edited without matching dir) fails the build with a clear error
referencing FR-011. A run-time mismatch (`.marque.toml [capco] version`
disagrees with `SCHEMA_VERSION`) is a fatal config-load error
(exit `65 EX_DATAERR`). Task T010 owns the build-time assertion; task
T023 owns the run-time check.

**`[corrections]` TOML schema**: the `[corrections]` table maps source
strings to replacement strings. Keys are matched as **exact byte-for-byte
substrings** inside marking candidate spans (never across a span
boundary); no canonicalization, no case-folding, no regex. Values are the
literal replacement bytes. Both keys and values must be valid UTF-8.
Example:

```toml
[corrections]
"SERCET" = "SECRET"
"NOFOR" = "NOFORN"
```

User corrections take precedence over any built-in rule match on the same
span (FR-009). Emitted `FixProposal`s from this table carry
`source = FixSource::CorrectionsMap`, `confidence = 1.0`, and
`migration_ref = Some("corrections-map")`. The map is compiled to a
perfect-hash (`phf`) or `rapidhash` table at config-load time for
O(1) lookup during scan.

**Source**: spec key entity *Configuration*; FR-007, FR-008, FR-009, FR-010,
FR-011.

## State transitions

The MVP has exactly one meaningful state transition:

```text
Diagnostic.fix: FixProposal ──(Engine::fix && confidence >= threshold)──▶ AppliedFix
```

`AppliedFix` is constructed once, in `Engine::fix`, and is immutable
thereafter. `FixProposal` is pure data with no state. `IsmAttributes`, `Span`,
`MarkingCandidate`, and `Configuration` have no internal state transitions —
they are constructed and consumed.
