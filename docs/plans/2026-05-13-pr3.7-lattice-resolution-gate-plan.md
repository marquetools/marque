### 1.3 Closure operator lives in `marque-scheme`, NOT `marque-rules`

D18 + T108c are explicit: the closure operator is **scheme-layer only**. It does NOT reference `marque-rules` types (`FixIntent` / `ReplacementIntent` / `FactRef`); fix-promotion semantics live in the engine's fix pipeline, not the closure operator. This preserves the workspace dependency invariant (Constitution VII: `marque-rules` depends on `marque-scheme`, never the reverse).

The mechanism: closure walks `closure_rules()` to Kleene fixpoint, applying each row's `cone` by direct mutation of `Self::Marking` via scheme-layer category-join operations. Each `TokenId` is added to its host category by joining with a singleton, dispatched through a scheme-provided `TokenId ‚Üí CategoryId` lookup.

### 1.4 `AuditNote` lives in `marque-rules`, parallel to `AppliedFix`

D19 A is explicit: `AuditNote` is in `marque-rules` alongside `AppliedFix`. This is correct under Constitution VII ‚Äî `marque-rules` already depends on `marque-scheme`, and `AuditNote.kind: AuditNoteKind` + `AuditNoteStructural` carry only `TokenId` / `Scope` / `Span` / `RuleId` / `Confidence` ‚Äî types that already exist in `marque-rules` or `marque-scheme`.

But there's a tension: the closure operator (`marque-scheme`) fires; the `AuditNote` (`marque-rules`) is produced by the engine. The engine is the converter ‚Äî it observes a closure firing (during `Engine::project` or wherever the call-site lands in PR 4) and constructs the `AuditNote` from the firing's `(ClosureRule, marking_state)` pair.

For **PR 3.7** specifically: the `AuditNote` type definition + sealing pattern + NDJSON shape ships here. The engine *construct-site* for `AuditNote` may defer to PR 4 alongside `Engine::project::closure()` wiring, because PR 4 owns the call-site. Or, if we can wire a thin emit path from `Engine::fix_inner` for closure-row severity surfaces (T108f, when severity ‚â• `Warn`), that lands here.

**Decision deferred to preflight reviewer**: does `AuditNote` *engine construct-site* land in PR 3.7 or PR 4? My current view is "types + sealing here, construct-site PR 4" to keep PR 3.7 from racing PR 7 on `Engine::fix_inner`. The trade-off is that PR 3.7's `AuditNote` is dead-code until PR 4 wires it ‚Äî which is intentional under D14 (trait surface stabilization) but means PR 3.7 reviewers can't observe an end-to-end audit_note line in NDJSON output.

### 1.5 `[closure_rules]` config section parsing (T108f)

The existing `.marque.toml` parser is in `marque-config`. T108f lands:

- New `[closure_rules]` table parsing (separate from `[rules]`)
- Same `Severity` enum surface
- `Severity::Fix` rejection at load time with new `ConfigError::InvalidClosureRuleSeverity { rule: String, hint: &'static str }` variant (per rust-preflight F4). Message: `"closure rule '{rule}' cannot use 'fix' severity; use 'warn' or 'error' (or 'off'/'info'/'suggest' for non-blocking surfaces)"`
- Per-row override mechanic identical to `[rules]`, but section-isolated (no fallthrough)

**TOML key quoting (per rust-preflight B4)**: `ClosureRule.name` values like `"capco/noforn-if-no-fdr"` contain `/`, which is not a legal unquoted-TOML-key character. `[closure_rules]` entries MUST use quoted form: `"capco/noforn-if-no-fdr" = "warn"`. The config parser docs comment will state this requirement explicitly, and one of the unit tests at `crates/config/tests/closure_severity.rs` will use the quoted form to pin the convention in test code.

This is mechanical ‚Äî no engine touch. Lives entirely in `crates/config/`.

### 1.5b `MARQUE_CLOSURE_RULES_*` env-var namespace (per architect-preflight R2)

The existing `[rules]` config supports env-var overrides via `MARQUE_RULES_<ID>=<severity>`. The `[closure_rules]` section needs a parallel namespace to maintain D19 B's section isolation:

- `MARQUE_CLOSURE_RULES_capco__noforn_if_no_fdr=warn` overrides the `"capco/noforn-if-no-fdr"` closure row
- Naming convention: `MARQUE_CLOSURE_RULES_<NAME>` where `<NAME>` is the row name with `/` replaced by `__` and other special characters lowercased (matches the established `MARQUE_RULES_*` convention for slash-containing rule IDs)
- Precedence ladder unchanged: CLI flags ‚Üí env vars ‚Üí `.marque.local.toml` ‚Üí `.marque.toml`
- A `[rules]` entry and a `[closure_rules]` entry with the same key string resolve independently ‚Äî section split is total. Same property applies to env vars: `MARQUE_RULES_X` is not consulted for closure-row resolution.

Unit tests at `crates/config/tests/closure_severity.rs` will pin the env-var override behavior parallel to the TOML override behavior.
